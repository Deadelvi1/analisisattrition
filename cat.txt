EMPLOYEE ATTRITION PREDICTION - DOKUMENTASI TEKNIS

1. Ringkasan Sistem
-------------------
Aplikasi ini memprediksi risiko pengunduran diri (attrition) karyawan menggunakan model Random Forest yang dibangun dengan scikit-learn. Hasil prediksi ditampilkan di halaman `/predict` bersama confidence score dan rule-based risk score. Backend menggunakan Flask, frontend menggunakan HTML/Jinja + CSS, dan deployment Railway memerlukan `PORT` serta `gunicorn`.

2. Arsitektur Teknis
--------------------
- Backend: Flask (`app.py`)
- Model: scikit-learn Random Forest dengan pipeline preprocessing
- Data: `employee_data.csv`
- Output model: `model/rf_model_tuning_latest.pkl`
- Metrics: `model/performance.json`
- Deployment: Railway / Gunicorn

3. Training dan Model Creation
------------------------------
- File utama: `model/modeling.py`
- Dataset dibaca dari `employee_data.csv`.
- Data target: kolom `Attrition` yang dikonversi ke integer.
- Data split: `train_test_split` 80/20 dengan `stratify=y` untuk menjaga distribusi kelas.

Fitur preprocessing:
- Numerik: `SimpleImputer(strategy='median')`, lalu `StandardScaler()`.
- Kategori: `SimpleImputer(strategy='most_frequent')`, lalu `OneHotEncoder(handle_unknown='ignore')`.
- Kategori memungkinkan model menerima nilai baru di runtime tanpa error.

Pipeline:
- `ColumnTransformer` untuk memisahkan numeric/categorical.
- `Pipeline([('prep', preprocessor), ('model', RandomForestClassifier(...))])`

Model tuning:
- `GridSearchCV` dengan:
  - `model__n_estimators`: [100, 200]
  - `model__max_depth`: [5, 10, None]
  - `cv=3`
  - `scoring='f1'`
- `class_weight='balanced'` untuk menangani ketidakseimbangan kelas attrition.

4. Evaluasi Model
-----------------
- Test Accuracy
- Precision
- Recall
- F1 Score
- `cross_val_score` untuk `cv_f1_mean` dan `cv_f1_std`
- `train_acc` dan `overfit_gap`

Output disimpan ke `model/performance.json` dengan format:
```
{
  "test": {"accuracy": ..., "f1": ..., "precision": ..., "recall": ...},
  "val": {"accuracy": ...},
  "overfit_gap": ..., 
  "cv_f1_mean": ..., 
  "cv_f1_std": ...
}
```

5. Backend `app.py` Detail
--------------------------
- `MODEL_PATH = 'model/rf_model_tuning_latest.pkl'`
- `PERF_PATH = 'model/performance.json'`
- App akan mencoba load model dan metrics saat startup.
- Jika `performance.json` tidak ada, app menggunakan fallback metrics hardcoded.
- Route `/predict`:
  - `GET`: render form input.
  - `POST`: `request.form.to_dict()` -> `pd.DataFrame([raw_data])` -> `model.predict(df)`.
  - `model.predict_proba(df)[0][1]` untuk `confidence` kelas `Resign`.
  - `compute_rule_risk(raw_data)` memberikan skor risiko tambahan.

6. Fungsi Rule-based Risk
-------------------------
Fungsi ini menilai risiko berdasarkan aturan bisnis HR:
- `EnvironmentSatisfaction <= 2` → +2
- `JobSatisfaction <= 2` → +2
- `OverTime` = yes → +2
- `WorkLifeBalance <= 2` → +1
- `YearsSinceLastPromotion >= 3` → +1
- `YearsAtCompany <= 1` → +1
- `DistanceFromHome > 20` → +1
- `NumCompaniesWorked >= 5` → +1
- `MonthlyIncome < 4500` → +1

Skor dijumlahkan dan dinormalisasi: `min(1.0, score / 12) * 100`.

7. Logika Output Prediksi
--------------------------
- `status_model`: hasil ML model (`Resign` jika pred == 1).
- `status_rule`: aturan bisnis (`Resign` jika `rule_risk >= 55`).
- `final_status` saat ini mengikuti hasil ML model.
- Jika `status_model` dan `status_rule` berbeda, sistem menambahkan penjelasan.
- `confidence_class` menentukan warna progress bar:
  - >= 70% → `success`
  - >= 40% → `warning`
  - < 40% → `danger`

8. Frontend Prediksi dan UI
---------------------------
- Form input meminta 14 fitur utama saja.
- Fitur lain diberikan `input type="hidden"` agar dataframe lengkap sesuai model.
- `loading-overlay` muncul saat submit, menonaktifkan tombol untuk menghindari klik ganda.
- Progress bar confidence menggunakan inline width dari `result.confidence`.
- `rule_risk` ditampilkan di card detail dengan bar terisi melalui JavaScript.
- Penilaian risiko ditampilkan dengan label:
  - Rendah (0-30%)
  - Sedang (30-70%)
  - Tinggi (70-100%)

9. Kenapa hasil bisa berbeda?
----------------------------
- Prediksi utama berasal dari model ML.
- Rule-based risk adalah pemeriksaan tambahan, bukan pengganti.
- Perbedaan dapat terjadi karena model belajar pola dari data, sementara aturan bisnis memeriksa kondisi tertentu.
- Ini normal: `rule_risk` membantu menjelaskan keputusan dan memberi sinyal yang lebih mudah dipahami.

10. Validitas dan Akurasi
-------------------------
- Parameter model disesuaikan untuk mengurangi overfitting.
- `class_weight='balanced'` membantu model menghitung bobot kelas minoritas.
- `F1 52.78%` menunjukkan performa moderat pada kelas target.
- Artinya model layak dipakai sebagai alat bantu HR, tetapi tidak sebagai keputusan tunggal.

11. Deployment Railway
----------------------
- `app.py` membaca `PORT` dari environment, jadi cocok untuk Railway.
- Tambahkan file `Procfile`:
  `web: gunicorn app:app`
- Pastikan `requirements.txt` berisi semua dependensi runtime.
- Railway akan menjalankan `pip install -r requirements.txt` lalu `gunicorn app:app`.

12. Catatan Tambahan
-------------------
- `model/modeling.py` menggunakan `mlflow` untuk tracking, tetapi runtime app tidak butuh `mlflow`.
- Jika model sudah terbuat, cukup pastikan `model/rf_model_tuning_latest.pkl` ada.
- Untuk deployment, masukkan `model/` ke dalam repository atau upload model ke artifact storage agar Railway dapat membacanya.
